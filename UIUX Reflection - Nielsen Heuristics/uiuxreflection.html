Norman + Nielsen

In the article on usability heuristics, Nielsen talks about matching the system being designed with the real world. This idea highly aligns with Norman's design concerns. Nielsen says that interfaces should use familiar language and conventions, as well as natural mapping, so that people can understand and remember how the interface works. This relates very closely to Norman's idea that users should be able to tell what actions are possible and build a correct understanding of how a design/system works. He mentions natural signals, mapping, signifiers and conceptual models as ways to achieve this and stresses that users fail when the system forces them to think in the machine's logic instead of their own. Nielsen also says something similar to this, mentioning natural mapping and mental models as important concepts.
There are several other areas of overlap between the two readings. One of the strongest of these is visibility of system status. Nielsen says that users should always be informed about what is going on through appropriate and timely feedback. Norman says nearly the same thing, mentioning that good feedback must be immediate, informative and appropriately prioritized. Some of the other areas of overlap are in the ideas of recognition rather than recall, error prevention and recovery, and aesthetic and minimalist design.

Micro-Project Heuristics Issues

The two heuristics that I believe my micro-project struggles with the most are #3 - User control and freedom, and #7 - Flexibility and efficiency of use. For #3, the user does not have any way to go back, undo, or redo an action. There is an inital starting menu but no pause menu or function. Once the game starts, the user is locked in with no way of pausing or exiting without closing the whole window. This can be especially frustrating if the user accidentaly presses the key to start the game. For #7, I didn't include any shortcuts or ways for experienced users to speed up their interactions. There are also no personalization or customization options available for the user either. The game always moves at the same speed and the same way. The only way for users to finish the game faster is through experience and memorization of the maze. The lack of these features could potentially make the game less enjoyable for users, especially those who are more experienced with games or want more control over their interactions.

Preliminary Sketchbook Interface Audit

The one heuristic that my sketchbook violates most clearly is #1 - Visibility of system status. Due to the fact that I haven't implemented any css styling, users are not provided with clear feedback about the system's state. Much of a user's ability to understand an interface comes from the way it is desgined - the way elements are arranged, how they are spaced out, how they look, and how they react when interacted with by the user. Right now, my sketchbook only utilizes the default html styling, which is functional but may not be as intuitive and optimal when it comes to visibility and feedback.

The Hardest Heuristic

I think the heuristic that is the hardest to implement honestly is #5 - Error prevention, due to both technical constraints and conceptual tension. Preventing errors before they happen sounds like a difficult task to accomplish, since there are so many different kinds of errors that could potentially occur and you never know what sort of error a user might encounter. It requires a lot of foresight and anticipation of user behavior but also observing and learning from user interactions, which can take a lot of time and effort. The technical constraints also make this difficult because of the expertise it takes to fix and prevent errors, as well as due to the fact thatit can be hard to implement features that prevent errors without making the interface more complex or less intuitive.
